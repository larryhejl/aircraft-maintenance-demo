Developer Guide

Architecture

The application is a single-page static web app with no build step, no frameworks, and no server-side dependencies.

Data Flow

initializeDashboard()
  ├── populateFilters(events)      → Fill dropdowns
  ├── filterEvents(events)         → Apply default filters
  ├── renderTable(filtered)        → Build data table
  ├── groupEventsByMonth(filtered) → Prepare chart data
  ├── renderChart(grouped)         → Draw bar chart
  └── renderStatusPie(filtered)    → Draw pie chart

Filter change (user interaction):
  ├── filterEvents(events)         → Re-apply filters
  ├── renderTable(filtered)        → Rebuild table
  ├── groupEventsByMonth(filtered) → Regroup
  ├── renderChart(grouped)         → Redraw bars
  └── renderStatusPie(filtered)    → Redraw pie

Test Contract

The test file must export window.runAllTests() returning:

{
  total: Number,
  passed: Number,
  failed: Number,
  coveragePercent: Number,
  results: [
    { name: String, status: "pass"|"fail", durationMs: Number, details?: String }
  ]
}
Convention: Tests that fail intentionally include a details field with a code improvement recommendation. This drives the "test → fix" demo cycle.

Known Limitations (Baseline)

  1. XSS risk — renderTable uses innerHTML without sanitization
  2. No input validation — parseEventDate returns Invalid Date for bad inputs instead of a safe fallback
  3. NaN keys — groupEventsByMonth produces "NaN-NaN" keys from invalid dates
  4. Unsorted filters — populateFilters inserts options in arbitrary Set-iteration order
  5. Non-determinism — filterEvents uses new Date() internally, making date-boundary behavior untestable

Extending the Application

Adding a New Filter

  1. Add a <select> element in the #controls section of index.html
  2. Update populateFilters() to fill the new dropdown
  3. Update filterEvents() to apply the new filter criteria
  4. Add corresponding unit tests in tests.js

Adding a New Chart Type

  1. Add a container <div> in the charts section
  2. Create a render function following the pattern of renderChart() or renderStatusPie()
  3. Call it from both initializeDashboard() and the filter-change handler
